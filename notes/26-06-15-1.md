## wity-scene → cinema-grade pipeline: ontological assessment

The question was whether our ontology has the foundational flexibility to compose into cinema-grade pipelines over time, compared to products like HyperFrames which have an inherent architectural ceiling.

---

### Why HyperFrames (and that class of tools) is capped

The ceiling isn't a feature gap — it's the rendering model itself. `HTML → Puppeteer → PNG → FFmpeg` bakes in constraints that cannot be evolved out of:

- Compositing happens inside the browser compositor — you cannot intercept passes or change blend operators below CSS
- Everything is sRGB — browser gamma is fixed, no linear light, no colorspace metadata on assets
- No render passes — one flat RGBA out, no beauty/diffuse/specular/shadow/depth separation
- 4K via Puppeteer is memory-hungry and designed for screenshots, not renders
- PBR is architecturally impossible from this stack

These are not fixable by adding features. The model is wrong at the foundation.

---

### Why our ontology is genuinely different

The core property that matters: `f(scene, t) → ComputedFrame` is a pure data transformation. The renderer is not part of the document model. This separation is what creates the ceiling difference.

Today ComputedFrame is consumed by Canvas 2D or HTML/CSS. But the ontology places no constraint on the consumer — the same scene document can drive:

- WebGPU compositor (blend modes, linear light)
- Arnold / RenderMan (PBR, ray tracing)
- OpenEXR writer (multi-channel passes)
- FFmpeg ACES chain (color science output)

---

### What each cinema requirement actually needs from us

**Multi-layer compositing with blend modes** — schema addition only. `<ws-layer blend-mode="add">`. Evaluator passes blendMode through, GPU compositor applies the operator. Low effort, no architectural change.

**Color science / ACES / LUT pipelines** — schema metadata + colorspace-aware renderer. `<wity-scene colorspace="ACEScg">`, `<ws-image colorspace="linear-sRGB">`, `<ws-layer grade="film-look.cube">`. Significant work but nothing ontologically blocking it.

**EXR sequences / massive asset pipelines** — `ws-image` already references assets by URL. Extend to `src-pattern="shot_042.%04d.exr" fps="24"`. `evaluate(scene, t)` resolves the frame index. The renderer handles EXR decode. Additive.

**Physically-based rendering** — requires a new element type (`ws-3d-object`) with material, lights, camera. `registerElementType` is open so this is additive. The evaluator dispatches these elements to a GPU render backend. Existing 2D elements are unaffected.

**Shot-level artistic control / sequence structure** — a wrapper level above the current scene. `<wity-sequence>` containing `<wity-shot>` containing `<wity-scene>`. Doesn't conflict with anything in scene-core — it's a document hierarchy extension.

**Multi-channel render passes** — ComputedFrame would need to support multi-channel output (beauty, diffuse, specular, depth, motion vectors). Extension to the evaluate() output shape, not a schema change.

---

### Effort table

| Capability | Effort | Ontological blocker |
|---|---|---|
| Blend modes on layers | Low | No |
| EXR frame sequences | Medium | No |
| Color management / ACES | High | No |
| Sequence / shot structure | Medium | No |
| Multi-channel render passes | High | No |
| PBR / ray tracing | Very high | No |
| Full cinema pipeline integration | Massive | No |

---

### Honest summary

The ontology has no ceiling because the document model and renderer are genuinely separated. Cinema-grade infrastructure — GPU compositing, OpenColorIO, EXR I/O, a PBR renderer — is real engineering that doesn't come for free. The difference from HyperFrames is that those are buildable on top of this model. On HyperFrames they are architecturally impossible.

The path exists. The ontology is right. The infrastructure has to grow around it.
